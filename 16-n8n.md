## 1. Overview & Learning Outcomes

n8n (pronounced "n-eight-n", short for *nodemation*) is an open-source, fair-code-licensed workflow automation tool — a self-hostable, source-available alternative to Zapier/Make with first-class developer features (Code nodes, sub-workflows, expressions, version control via JSON, and a native AI Agent node built on LangChain). It runs on Node.js and stores executions in a database (SQLite by default; Postgres in production).

For a QA-leaning engineer, n8n is the fastest way to:

- Glue together ticketing systems, chat, calendars, and test runners.
- Build "AI sidekicks" — Slack-triggered triage bots, daily test-result digests, RAG over your test docs.
- Prototype LLM workflows visually before committing to code in LangChain/LangGraph.

By the end of this subject you should be able to:

- Stand up self-hosted n8n with Docker, Postgres, and reverse proxy.
- Compose workflows using triggers, core nodes, and Code/Function nodes with idiomatic JS or Python expressions.
- Wire credentials securely, including OAuth2 flows.
- Build an AI Agent workflow with tool calls, vector store retrieval, and structured output.
- Version, test, and deploy workflows like code.
- Choose between n8n, LangChain, and LangGraph for a given problem, and explain the trade-offs.

## 3. Topics — Explanations with Examples

### 3.1 Cloud vs self-hosted

**Concept.** n8n Cloud is a managed offering with usage-based pricing. Self-hosting is free under the **Sustainable Use License** (fair-code) for internal/personal use; commercial resale or hosting-as-a-service is restricted. For learning and most QA use cases, self-host.

```bash
# Quickest path — npx run, ephemeral
npx n8n

# Docker, persistent volume
docker volume create n8n_data
docker run -it --rm \
  --name n8n \
  -p 5678:5678 \
  -v n8n_data:/home/node/.n8n \
  -e N8N_SECURE_COOKIE=false \
  docker.n8n.io/n8nio/n8n
```

Open `http://localhost:5678`, set up the owner account, you're in.

### 3.2 Workflow anatomy

A workflow is a directed graph of **nodes** connected by **connections**. Execution starts from a **trigger** node (one or more), runs each downstream node, and exits.

Key UI panes:

- **Editor** — the canvas, node parameters on the right.
- **Executions** — history, with input/output per node, replayable.
- **Credentials** — encrypted secrets per integration.
- **Variables** — workspace-wide string vars (great for env names, base URLs).
- **Projects** — multi-tenant grouping; useful for separating dev/prod or teams.

### 3.3 The item model — the most important mental model

**Concept.** Every node receives **and produces an array of items**. An item is `{ json: {...}, binary?: {...}, pairedItem?: {...} }`. Most "loops" in n8n are implicit — a node with 50 items runs *once but processes all 50*. Only specific nodes (like SplitInBatches or sub-workflow loops) actually iterate.

This trips everyone up at first. Internalize it:

- Five rows from a database → 5 items.
- An IF node splits into two branches; each branch carries a subset of items.
- Merging back may use `Merge → Append`, `Combine`, or `Wait`.

### 3.4 Triggers

Common triggers:

- **Manual** — for testing.
- **Schedule** — cron-like (`*/15 * * * *`, or "every Monday 9am").
- **Webhook** — exposes a URL; n8n becomes an HTTP endpoint.
- **App-specific** — Gmail new email, GitHub issue, Slack message, etc.

```text
Webhook URL pattern (production):
https://n8n.example.com/webhook/<path>
Test URL (only active while editor is open):
https://n8n.example.com/webhook-test/<path>
```

Webhook security: validate `X-Hub-Signature` (GitHub), `X-Slack-Signature`, or your own HMAC header in a Code node before doing real work.

### 3.5 Core node patterns

- **HTTP Request** — the universal "any REST API" node. Supports auth, query/headers/body, pagination, batching.
- **Set** — assign or rename fields; cleaner than Code for simple shaping.
- **IF / Switch** — branching by condition (single vs multi).
- **Merge** — append, keep keys, multiplex; choose mode carefully.
- **SplitInBatches** — *the* loop primitive when you need to genuinely paginate (e.g., "process 100 items at a time, sleep 1s between").
- **Wait** — pause until time, webhook, or duration.

**Example — paginate a REST API.**
```text
Schedule (daily 06:00)
  → HTTP Request (GET /jobs?page=1)
  → SplitInBatches (size=1)
    → HTTP Request (GET /jobs/{{ $json.id }})
    → Postgres (insert)
  → IF (next page exists)
    → loop back to SplitInBatches
```

### 3.6 Expressions

Expressions are JS-flavored, sandboxed, and start with `{{` and end with `}}`.

```text
{{ $json.user.email }}
{{ $json.items.length }}
{{ $now.format('yyyy-MM-dd') }}
{{ $node["HTTP Request"].json.id }}
{{ $if($json.score > 0.8, 'high', 'low') }}
{{ DateTime.now().minus({ days: 7 }).toISO() }}
```

`$json` is the *current* item; `$node["..."].json` reaches sideways. Built-ins include `$now`, `$today`, `DateTime` (Luxon), `$if`, `$itemIndex`, `$workflow`.

### 3.7 Code node

When expressions get long, drop into a Code node. It supports JS (default) and Python (Pyodide).

```js
// Run Once for All Items
const grouped = {};
for (const item of items) {
  const k = item.json.country;
  grouped[k] ||= [];
  grouped[k].push(item.json);
}
return Object.entries(grouped).map(([country, rows]) => ({ json: { country, count: rows.length, rows } }));
```

```python
# Python (Pyodide), Run Once for Each Item
import statistics
scores = item["json"]["scores"]
return {"json": {"avg": statistics.mean(scores), "n": len(scores)}}
```

`$helpers.httpRequest` lets the Code node make calls without dropping back to an HTTP node.

### 3.8 Binary data

Files travel on `item.binary.<key>` with `data` (base64), `mimeType`, `fileName`, `fileExtension`. The HTTP Request node, Read/Write Binary File, S3, and AI nodes all speak this convention.

```text
Webhook (multipart upload)
  → AI Agent (vision capable)
  → Set { description: $json.text }
  → Slack (post message with file)
```

### 3.9 Credentials and OAuth2

Credentials are encrypted at rest using `N8N_ENCRYPTION_KEY` (set this and back it up — losing it bricks your credentials). For OAuth2:

1. Register an OAuth app (Google/Slack/Microsoft).
2. Authorized redirect: `https://<your-n8n>/rest/oauth2-credential/callback`.
3. Add credential, click "Connect my account", complete the consent flow.

For env-driven secrets in self-host: `N8N_SECURE_COOKIE`, `N8N_BASIC_AUTH_USER`, OAuth client IDs/secrets all set via env. Never paste secrets into HTTP nodes.

### 3.10 Error handling

Per-node:

- **Continue On Fail** — keep going, push the error onto the item.
- **Retry On Fail** — retries with backoff.

Workflow-wide:

- Mark a workflow as an **Error Workflow** in another workflow's Settings → Error Workflow. The error workflow receives the failing execution metadata.

```text
Error Workflow:
  Trigger (Error)
    → Slack (post to #ops)
    → Linear (create incident issue)
```

### 3.11 Sub-workflows

Use **Execute Workflow** to call another workflow as a function:

- DRY — extract "send notification with retry" once, call it from 10 places.
- Testability — test the sub-workflow in isolation.
- Versioning — sub-workflows are referenced by ID, so changes propagate.

### 3.12 The AI Agent node

n8n's flagship AI feature. Internally, it's a wrapper around LangChain agents with tool-calling.

Key inputs:

- **Chat Model** — pick OpenAI, Anthropic, Google, Ollama (Self-hosted), Bedrock, etc.
- **Memory** — Window Buffer / Conversation Buffer / Postgres / Redis.
- **Tools** — any node added downstream becomes a "tool the agent can call". Built-ins: HTTP Request Tool, Calculator, Wikipedia, SerpAPI, Vector Store retrievers.
- **System message** — agent persona/instructions.

**Example shape — Slack triage bot.**
```text
Slack Trigger (new message in #qa-bugs)
  → AI Agent
        Chat Model: OpenAI gpt-4o-mini
        Memory: Window Buffer (k=10)
        Tools:
          - Vector Store Retriever (pgvector — past bugs corpus)
          - HTTP Request Tool (Linear API: search_issues)
          - Code Tool (return reproduction template)
        System: "You are a QA triage assistant…"
  → Slack (reply in thread)
```

### 3.13 LLM provider nodes

n8n exposes one node per provider plus a generic **Basic LLM Chain** node for prompt-only usage. Local LLMs work via the **Ollama Chat Model** node — point to your Ollama server (default `http://host.docker.internal:11434` from inside the n8n container).

### 3.14 Document Loaders, Splitters, Embeddings

Mirror LangChain's primitives, but as visual nodes:

- **Document Loader** — Default Data Loader, GitHub, Web, S3, etc.
- **Text Splitter** — Recursive Character, Token, Markdown.
- **Embeddings** — OpenAI, Cohere, Hugging Face, Ollama.

**Example — "ingest a Markdown corpus into pgvector".**
```text
Manual Trigger
  → Read Files (path: ./docs/*.md)
  → Document Loader (Default Data Loader)
  → Recursive Character Text Splitter (chunk=1000, overlap=200)
  → Embeddings (OpenAI text-embedding-3-small)
  → Vector Store (pgvector — table: docs)
```

### 3.15 RAG workflow

The retrieval flow then uses a **Vector Store Retriever** as a tool to the AI Agent (or attached to a basic chain). Hybrid search and re-ranking are typically done outside n8n (file 13) or by chaining a re-ranker node where supported.

### 3.16 Output parsers and structured output

For workflows that need JSON downstream:

- **Structured Output Parser** — provide a JSON schema; node validates and casts.
- **Auto-fixing Parser** — re-prompts the LLM on parse failure.

```json
{
  "type": "object",
  "properties": {
    "category": { "type": "string", "enum": ["bug","feature","question"] },
    "severity": { "type": "string", "enum": ["S1","S2","S3","S4"] },
    "summary":  { "type": "string" }
  },
  "required": ["category","severity","summary"]
}
```

### 3.17 Self-hosting (production)

```yaml
# docker-compose.yml (excerpt)
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: n8n
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: n8n
    volumes: [pgdata:/var/lib/postgresql/data]

  n8n:
    image: docker.n8n.io/n8nio/n8n:latest
    depends_on: [postgres]
    ports: ["5678:5678"]
    environment:
      DB_TYPE: postgresdb
      DB_POSTGRESDB_HOST: postgres
      DB_POSTGRESDB_USER: n8n
      DB_POSTGRESDB_PASSWORD: ${POSTGRES_PASSWORD}
      DB_POSTGRESDB_DATABASE: n8n
      N8N_HOST: n8n.example.com
      N8N_PROTOCOL: https
      WEBHOOK_URL: https://n8n.example.com/
      N8N_ENCRYPTION_KEY: ${N8N_ENCRYPTION_KEY}
      EXECUTIONS_MODE: regular            # or 'queue' for workers
      GENERIC_TIMEZONE: Asia/Kolkata
    volumes: [n8n_data:/home/node/.n8n]

volumes: { pgdata: {}, n8n_data: {} }
```

Front with Caddy/Traefik/nginx for TLS. Set `N8N_ENCRYPTION_KEY` once and never lose it.

### 3.18 Queue mode (scaling)

For high throughput, switch to `EXECUTIONS_MODE=queue` with Redis and run separate **worker** containers. The main container handles UI/triggers; workers run executions.

```text
            ┌── n8n main (UI, triggers, webhooks)
Redis ◀──┐  │
         └──┼── n8n worker 1
            ├── n8n worker 2
            └── n8n worker N
Postgres ◀──┘
```

### 3.19 Versioning workflows in Git

Workflows are JSON. Treat them like code.

```bash
# Export every workflow to disk
n8n export:workflow --all --output=./workflows

# Import on a fresh instance
n8n import:workflow --separate --input=./workflows

# Useful for env promotion (dev → staging → prod)
```

Pair with file 15: branch per change, PR review, semantic commit messages, CI deploys.

### 3.20 Testing workflows

Strategies:

- **Pinned data.** Right-click a node → Pin Data — locks the input/output so tests are reproducible during development.
- **Test Workflow button.** Runs the whole workflow on the manual trigger.
- **Evaluation runs (n8n AI features).** Loop a dataset of inputs through an AI Agent and compare against expected outputs.
- **Sub-workflow tests.** Build a "test runner" workflow that calls a sub-workflow with a fixture and asserts via IF + Stop and Error.
- **External tests.** Hit the production webhook URL from pytest/Playwright, then poll the executions API for `success`/`failed`.

```python
# Smoke-test a webhook from pytest
import requests, time, os
BASE = os.environ["N8N_BASE"]
KEY  = os.environ["N8N_API_KEY"]

def test_intake_webhook():
    r = requests.post(f"{BASE}/webhook/intake", json={"x": 1}, timeout=10)
    assert r.status_code == 200
    # poll executions API
    headers = {"X-N8N-API-KEY": KEY}
    for _ in range(10):
        execs = requests.get(f"{BASE}/api/v1/executions?workflowId=42&limit=1", headers=headers).json()
        if execs["data"] and execs["data"][0]["finished"]:
            assert execs["data"][0]["status"] == "success"
            return
        time.sleep(1)
    raise AssertionError("execution did not finish")
```

### 3.21 Observability

- **Executions DB** — every run, with full payload (configurable retention via `EXECUTIONS_DATA_PRUNE`).
- **Logs** — `N8N_LOG_LEVEL=debug` for development; pipe to Loki/Datadog in prod.
- **Metrics** — `N8N_METRICS=true` exposes Prometheus metrics on `/metrics`.
- **Error workflows** — see §3.10; pair with PagerDuty/Opsgenie for real incident routing.

### 3.22 When to pick n8n vs LangChain vs LangGraph

| Need                                                | Reach for       |
|------------------------------------------------------|-----------------|
| Visual flow, business stakeholders edit it           | n8n             |
| Heavy code, custom retrievers/parsers/eval pipelines | LangChain (18)  |
| Stateful multi-agent, cycles, human-in-the-loop      | LangGraph (17)  |
| Simple cron/webhook glue, no AI                      | n8n             |
| Production AI app with tracing & ops                 | LangChain + LangGraph + LangSmith |

n8n is excellent for *integration-heavy* AI workflows and rapid prototyping. When you outgrow the canvas — usually around complex state, parallelism, or evaluation — graduate the AI parts to LangGraph (file 17) and call into them from n8n via HTTP.

## 4. Exercises (Hands-On)

### Beginner

- **A1.** Self-host n8n with Docker + named volume. Set `N8N_ENCRYPTION_KEY`. Create the owner account.
  - *Acceptance:* You can stop and start the container without losing credentials.
- **A2.** Build *Hello, Webhook*: Webhook → Set → Respond to Webhook with `{ ok: true, hello: $json.name }`. Hit it with curl.
- **A3.** Build *Daily weather to Slack*: Schedule (07:00) → HTTP Request (open-meteo) → Format with Set → Slack.

### Intermediate

- **B1.** GitHub PR triage: GitHub Trigger (PR opened) → IF (title contains "[WIP]") → branch into Slack + Linear notifications using sub-workflows.
- **B2.** Pagination + dedupe: pull all pages from a public REST API into Postgres with SplitInBatches and an IF on a "next cursor" field. Idempotent re-run.
- **B3.** Error workflow: deliberately crash a workflow (HTTP 500); route the error to Slack #ops with execution URL.
- **B4.** Credentials & OAuth2: connect Google Drive; list new files in a folder; on add, transcribe with OpenAI Whisper and email a summary.

### Advanced

- **C1.** RAG over your test docs: ingest `*.md` from a Git repo into pgvector via Document Loader + Recursive Splitter + Embeddings. Build an AI Agent flow (Slack trigger) that answers questions about your test strategy with citations.
- **C2.** AI Agent triage bot: tools = (Linear API search, Vector Retriever, HTTP probe of CI). Output via Structured Output Parser. Post a Slack thread with severity, owner, and link.
- **C3.** Workflow-as-code: export all workflows; commit to Git with a CI pipeline that imports into a staging instance via `n8n import:workflow`.
- **C4.** Queue-mode upgrade: convert your Compose stack to `EXECUTIONS_MODE=queue` with Redis and 2 workers; run a load test (Artillery) hitting a webhook and watch worker distribution.

### Capstone — "QA Operations Hub"

Build a self-hosted n8n that powers your QA team's daily ops:

1. **Daily 08:00 digest** of failed CI builds (GitHub Actions API) summarized by an LLM and posted to Slack.
2. **Slack-triggered triage agent** with pgvector RAG over your runbooks + Linear tools.
3. **PR-opened workflow** that posts a checklist comment and creates a Linear sub-task if the diff touches `src/test/`.
4. **Error workflow** that opens a PagerDuty incident on prod failures.
5. **Workflow JSON** versioned in Git; CI deploys via `n8n import:workflow`; smoke tests in pytest.
6. README with architecture diagram, env-var matrix, and a runbook entry.
